# GB28181 livestream transport

**GB/T 28181** — PRC national standard *"Technical Specification for Information Transmission, Swap and Control of Video Surveillance Network"* — is China's mandatory interoperability standard for integrating video-surveillance devices (including drones) with provincial / municipal surveillance platforms. The DJI device registers with a cloud-hosted SIP server as a GB28181 agent and pushes the video stream over the standard's transport (PS-over-RTP).

See [`README.md`](README.md) for the device-support matrix across all four protocols and scope rules. MQTT signaling to start / stop / qualify a stream lives in [`../mqtt/dock-to-cloud/services/live_start_push.md`](../mqtt/dock-to-cloud/services/live_start_push.md).

---

## 1. MQTT selector

- `url_type` = **`3`** in all in-scope MQTT `live_start_push` enums (Dock 2, Dock 3, RC Plus 2 Enterprise, RC Pro Enterprise).

## 2. `url` shape

DJI source description (verbatim, v1.15): `serverIP&serverPort&serverID&agentID&agentPassword&localPort&channel`.

DJI source example (verbatim, v1.15):

```
serverIP=192.168.1.1&serverPort=8080&serverID=34000000000000000000&agentID=300000000010000000000&agentPassword=0000000&localPort=7060&channel=340000000000000000000
```

Shape: URL-encoded `key=value` pairs joined by `&`. **Not** a standard URI; the `url` field contains the raw query string alone (no `?`, no scheme).

| Key | Meaning | GB28181 term |
|---|---|---|
| `serverIP` | SIP-server (GB28181 platform) IP address. | SIP server address |
| `serverPort` | SIP-server port. | SIP server port |
| `serverID` | SIP-server identifier (20-digit GB28181 ID). | 上级平台ID / Superior-platform ID |
| `agentID` | The device's own GB28181 ID. | 设备ID / Device ID (20 digits) |
| `agentPassword` | Password the device uses to register with the SIP server. | 注册密码 / Registration password |
| `localPort` | Local SIP port the agent listens on (SIP registration response traffic). | Local SIP port |
| `channel` | Channel ID — which camera/gimbal channel on the device is pushing. | 通道ID / Channel ID (20 digits) |

GB28181 uses 20-digit numeric identifiers assembled from administrative region + device type + sequence per the national specification. DJI treats these as opaque strings; the cloud is responsible for allocating them according to GB28181 rules before issuing the `live_start_push`.

### 2.1 Wire flow (external to DJI)

After the device receives `live_start_push`:

1. **Register** — the device sends a SIP `REGISTER` to `serverIP:serverPort` using `agentID` + `agentPassword`, listening locally on `localPort`.
2. **Catalog** — the SIP server (cloud) queries the device catalog via a `MESSAGE` with `<Query><CmdType>Catalog</CmdType></Query>`; the device responds with the channel listing.
3. **Invite** — the SIP server sends `INVITE` targeting `channel`; the SDP body negotiates the RTP port for the media stream.
4. **Stream** — the device pushes an H.264 (or H.265) stream multiplexed as **MPEG-PS over RTP** to the SDP-agreed port.

The cloud must terminate all four steps. `url_type = 3` delivers only the registration parameters; everything past the register is standard GB28181 and not DJI-specific.

## 4. Cloud-side responsibility

The cloud must host a **GB28181-compliant SIP server + Media Stream server**. Common implementations:

- [WVP-Platform](https://github.com/648540858/wvp-GB28181-pro) — open-source GB28181 platform in Java/Spring.
- [SRS](https://github.com/ossrs/srs) (≥ 5.0) — supports GB28181 ingest plus RTMP / HLS / WebRTC re-publish.
- [ZLMediaKit](https://github.com/ZLMediaKit/ZLMediaKit) — media-server library with GB28181 plugin.
- Commercial Chinese surveillance platforms: 华为视频云 (Huawei), 海康 iSecure Center (Hikvision), 大华 DSS (Dahua).

**Deployment prerequisites** beyond what the DJI `url` delivers:

- The cloud must assign the device a valid 20-digit `agentID` in accordance with the operator's GB28181 ID allocation scheme. Mis-formatted IDs cause the SIP registration to fail.
- The cloud's SIP server must be reachable from the device's network (including any NAT traversal needs — GB28181 predates STUN/TURN and typically relies on public IP or VPN).
- The cloud must pre-configure a device record with a matching `agentPassword` or accept the password on first registration depending on implementation.

## 5. Worked example — complete `live_start_push` payload

MQTT message (Cloud → Device on `thing/product/{gateway_sn}/services`):

```json
{
  "bid": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxx",
  "data": {
    "url_type": 3,
    "url": "serverIP=192.168.1.1&serverPort=8080&serverID=34000000000000000000&agentID=300000000010000000000&agentPassword=0000000&localPort=7060&channel=340000000000000000000",
    "video_id": "1ZNDH1D0010098/39-0-7/normal-0",
    "video_quality": 0
  },
  "tid": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxx",
  "timestamp": 1654070968655,
  "method": "live_start_push"
}
```

The device parses the seven kv pairs, opens a local SIP listener on `localPort = 7060`, and sends a `REGISTER` to `192.168.1.1:8080` with its `agentID` + `agentPassword`. The cloud's SIP server accepts the registration and follows up with `Catalog` + `INVITE` per §2.1.

## 6. External references

- **GB/T 28181-2016** — current PRC national standard. PDF is paywalled via the SAC; translations circulate on Chinese technical forums. Superseded GB/T 28181-2011 versions of key sections differ in ID digit scheme and SIP profile.
- [Canonical English summary](https://en.wikipedia.org/wiki/GB/T_28181) — Wikipedia overview, updated irregularly.
- SIP-family dependencies — [RFC 3261](https://www.rfc-editor.org/rfc/rfc3261) (SIP core), [RFC 3550](https://www.rfc-editor.org/rfc/rfc3550) (RTP), [ITU-T T.140](https://www.itu.int/rec/T-REC-T.140) (text-over-RTP).
